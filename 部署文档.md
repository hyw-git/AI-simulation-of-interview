# AI 职场领航员部署文档

## 1. 部署目标

这份文档的重点不是单纯告诉你“怎么起服务”，而是说明项目当前如何在本地稳定运行，以及后续如何一步一步迁移到服务器。现阶段的目标是先把本地 Docker 环境打通，保证演示、调试和交付都没有问题；等业务和运行方式稳定下来之后，再把前后端、数据库和静态资源逐步迁到服务器上，形成更接近生产环境的部署方案。

## 2. 当前部署方式

当前版本使用本地 Docker Compose 进行部署。数据库、后端服务和前端开发环境都围绕同一套代码和配置运行，这种方式最大的好处是简单直接：数据库初始化脚本可以自动执行，环境变量可以统一管理，前后端也能在同一台机器上完成联调。对于课程答辩、功能演示和开发阶段来说，这种部署方式足够稳定，也更容易排查问题。

## 3. 当前本地运行方法

在开始之前，需要确认本地已经安装 Docker、Docker Compose 和 Node.js 18 以上版本。如果系统需要调用大模型能力，还要准备好 `.env.llm` 配置文件。准备完成后，只要在项目根目录执行 `docker-compose up --build`，数据库和后端就会一并启动，启动过程中还会自动执行 `db/schema.sql` 和 `db/seed.sql`，因此不需要额外手工导入数据。

前端则可以单独在本地开发模式下运行。进入 `frontend` 目录后执行 `npm install` 和 `npm run dev`，浏览器默认访问地址是 `http://localhost:5173`。这样的结构比较适合开发期，因为前端可以热更新，后端和数据库则保持稳定运行，调试效率会高很多。

## 4. 当前运行检查

服务起来之后，建议先访问后端健康检查接口 `http://localhost:8000/health`，确认数据库和后端已经正常连接。接着再打开首页、题库页和面试页，分别检查统计卡片、题目浏览、收藏功能以及流式会话是否可用。只要这几个关键页面都能正常工作，就说明当前本地环境已经具备交付和演示条件。

## 5. 当前配置说明

LLM 相关配置通过 `.env.llm` 管理，模板文件可以从 `.env.llm.example` 复制得到。这里最关键的是 `OPENAI_API_KEY`、`OPENAI_BASE_URL` 和 `OPENAI_MODEL` 这几个变量；如果暂时没有配置 API Key，系统会自动降级到规则模板和启发式逻辑，所以即使在离线或受限环境中也依然可以完成基本演示。数据路径方面，题库文件默认放在 `data/questions/`，简历文件默认放在 `data/resumes/`，向量索引缓存则写入后端缓存目录，便于本地测试和复用。

## 6. 后续服务器部署计划

当前项目虽然能在本地通过 Docker 跑通，但它的后续目标并不止于此。第一步建议先迁移到一台单机服务器，比如云服务器或校内服务器，在那台机器上安装 Docker，然后把后端、数据库和反向代理放在同一环境里，用 Docker Compose 直接启动。这一阶段的重点不是拆分架构，而是先把“服务器上能跑起来”这件事做稳。

当单机部署稳定之后，就可以进入第二阶段：引入 Nginx 作为统一入口，把前端静态资源交给它托管，同时将 `/api` 和 `/ws` 之类的请求反向代理到后端服务。这样做的目的，一方面是让访问入口统一，另一方面也是为 HTTPS 和域名访问做准备。

第三阶段的重点是配置和数据分离。到这一阶段时，数据库连接、模型密钥和其他环境变量都不应该再硬编码在镜像里，而应该统一交给服务器环境管理；简历文件、题库索引和日志也应该挂载到持久化目录，避免容器重建后丢失数据。

最后一步才是运维增强，包括把评估、语音转写和向量构建这类耗时操作拆分到后台任务中，补上日志收集、监控、告警和数据库备份策略。如果未来要把项目长期运行下去，这一层是必须补的，但它更适合作为后续迭代，而不是当前阶段的前置条件。

## 7. 交付检查项

在交付前，至少要确认首页能正常显示统计卡片和图表，题库页可以正常浏览和收藏，面试页可以正常发起会话并接收流式回复，面试结束后可以生成报告并进入历史记录，最后再检查 `/health` 接口是否正常返回。只要这些关键点都能通过，当前版本就具备了较好的演示和答辩条件。
